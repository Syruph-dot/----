# 性能优化方案

## 目标
- 在不改变玩法、碰撞语义、技能时序和 AI 决策结果的前提下，降低主循环 CPU 开销、canvas 开销和 GC 压力。
- 先做低风险、高收益的结构优化，再考虑局部渲染细化。

## 实施顺序
1. 实体缓存与就地压缩
2. AI 决策上下文复用
3. 静态背景与分隔带缓存
4. 受控的碰撞广义裁剪
5. 末端的子弹渲染细化

## 1. 实体缓存与就地压缩
- 目标：消除 getEnemies() / getBullets() 的重复 filter 分配，以及 update 末尾对 bullets / enemies 的整数组重建。
- 做法：在 Game 内维护按 side 和按活跃状态的缓存视图，或者在帧末使用原地压缩替代反复 filter。
- 预期收益：高，直接减少扫描次数和 GC 压力。
- 回滚策略：保留当前 filter 路径作为回退实现，必要时按 flag 切回。
- 验证点：敌机清理、子弹清理、Player 炸弹、WaveSystem 限流、AI 取数结果必须与当前一致。

## 2. AI 决策上下文复用
- 目标：减少每个决策 tick 内对同一批子弹和敌机的重复扫描。
- 做法：在 makeDecision() 内一次性收集当前危险子弹、敌机摘要、玩家中心和屏幕边界，再把缓存结果传给威胁计算和候选评分；避免 buildContext() 与 chooseMovementPlan() 之间重复取数。
- 预期收益：高，尤其是困难模式和弹幕密集时。
- 回滚策略：保留现有函数边界，必要时恢复为当前逐步扫描版本。
- 验证点：AI 技能请求顺序、移动方向、蓄力/炸弹触发条件和训练数据字段都不能变化。

## 3. 静态背景与分隔带缓存
- 目标：减少每帧重复绘制网格、分隔带和固定 UI 装饰的 canvas 成本。
- 做法：把不随战斗变化的背景层预渲染成离屏 canvas 或静态缓存层；保留扫描线、血量条和数值文本为动态层。
- 预期收益：中等，主要降低持续性的绘制和渐变创建成本。
- 回滚策略：任何视觉异常都可以直接切回当前逐帧绘制方式。
- 验证点：左右战区、分隔带高亮、扫描线动画和 HUD 显示要保持一致。

## 4. 受控的碰撞广义裁剪
- 目标：在不改变命中结果的前提下，减少 checkCollisions() 和 updateExpandingFields() 的无效比较。
- 做法：优先用 side、类型和粗边界做早退，再考虑为子弹和敌机建立轻量级空间分桶；保持精确碰撞检测仍然作为最终判定。
- 预期收益：中到高，取决于弹幕密度和敌机数量。
- 回滚策略：保留当前精确判定函数，任何新裁剪层都必须可单独关闭。
- 验证点：子弹命中、小怪爆炸、Boss 受伤、力场消弹、传送和连击变化必须保持完全一致。

## 5. 末端的子弹渲染细化
- 目标：在实体很多时降低每个子弹的绘制成本。
- 做法：只在确认瓶颈后再考虑简化阴影、渐变、路径绘制或为某些稳定外观的子弹使用更轻量的绘制分支。
- 预期收益：中等，主要在弹幕密度很高时体现。
- 回滚策略：保留现有渲染分支，必要时只恢复特效，不动逻辑。
- 验证点：不同子弹类型的可读性、转移提示、预警提示和激光效果不能被削弱到影响玩法判断。

## 风险控制
- 不改碰撞顺序，不改技能释放时序，不改 AI 决策规则，不改计分逻辑。
- 所有优化先做局部、可回退、可对照的改动，避免一次性重构主循环。
- 每一步都要做功能回归：正常开局、双人模式、AI 模式、炸弹、连击、Boss、技能 2/3/4、传送弹幕和游戏结束流程。

## 建议验证指标
- 帧时间：平均值、p95、p99。
- AI 决策开销：单次决策耗时和每秒调用次数。
- GC 压力：高频战斗场景下的分配量和暂停次数。
- 语义回归：命中数、击杀数、Boss 生成数、炸弹效果和胜负结果。

## 结论
- Phase 1 已完成，当前只给出报告与方案。
- 等待用户明确允许后，再按上面的顺序进入最小化实现。
