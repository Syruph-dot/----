# 《东方梦时空》AI 实现方式调研

## 结论

公开资料和 ReC98 逆向资料指向同一个结论：PoDD 的 CPU 对手不是学习型 AI，更像是“规则脚本 + 时间表 + 资源管理”的组合。

它的核心不是“推理出最优走位”，而是按预先设定的节奏执行这些动作：

- 在某些时间点故意吃到固定次数的伤害。
- 根据关卡、难度和当前对局阶段调整这些受击时间。
- 在可能躲不开的时候优先开炸弹或超载/蓄力保护。
- 在满能量时强制蓄满并立刻释放。
- 某些特殊角色/特殊回合会改成纯生存逻辑，而不是固定受击逻辑。

## 公开资料能确认的行为特征

### 1. AI 的主逻辑是“定时受击”

MaribelHearn Wiki 的高级机制页明确写到，AI 会在特定时间点“intentional hits”，也就是故意受击；这些时间会随着 stage 和 difficulty 变化，且每一关的节奏不同。

对实现层面的含义是：

- AI 不是逐帧搜索最优动作。
- 它更像是一个阶段机，内部维护“什么时候该挨打、什么时候该继续输出”的表。
- 受击时间是关卡设计的一部分，不是临场学习出来的。

### 2. 伤害量和时长会随对局进度变化

同一份资料还提到：AI 的受击伤害不是固定常量，而是会随着时间推进而变化；另外，较低关卡阶段和较低难度下，CPU 会承受更高的首击伤害或更短的间隔，以便控制整场对局时长。

这意味着实现上至少包含两层参数：

- 一层是“这局/这关允许打多久”。
- 一层是“每次被打时要掉多少血”。

### 3. 资源管理是防守逻辑的一部分

同一资料还说明：如果 CPU 发现自己即将吃到躲不开的伤害，会优先炸弹；如果没炸弹但能量条满了，就会先开 hyper/charge 来保命。若正在蓄力且马上要被击中，它会立刻释放蓄力，以利用短暂无敌帧。

这类逻辑更像“生存策略表”，不是“攻击型策略网络”。

### 4. 某些角色/回合有特例

资料里提到，Yumemi 的第一回合是特殊例外：她没有常规的计划性受击，而是尽可能存活；AI 会优先用 bombs / charge 维持生存。这个例外进一步说明，PoDD 的 CPU 行为是“分角色、分回合、分阶段”的脚本系统。

## 从源码/逆向能看到的实现方向

ReC98 的 TH03 代码里能看到和 CPU / demo 相关的结构字段，例如 `is_cpu`、`cpu_safety_frames`、`cpu_charge_at_avail_ring`、`cpu_frame`、`cpu_dodge_strategy` 等。这些字段不是神经网络参数，而是 CPU 行为状态机和阈值控制用的数据。

同时，`start_demo()` 会把双方都标成 CPU，并加载固定的 demo 配对，这也符合“预设脚本对战/回放”的思路，而不是在线训练模型。

## 可以把它概括成什么实现模式

如果把 PoDD 的 AI 拆成现代术语，大致可以概括成：

1. 关卡/难度驱动的状态机。
2. 规则驱动的定时受击脚本。
3. 资源阈值驱动的保命逻辑。
4. 角色特例和回合特例。
5. 少量随机数用于让行为不完全死板，但不改变整体骨架。

它不是“训练出来的策略”，而是“按设计图表演的对手”。

## 对本项目的参考价值

如果你想做“古早但贴合游戏实际”的 AI，PoDD 的价值在于两点：

- 它证明了弹幕对战 AI 可以先用脚本/阈值/阶段表做得很像样。
- 它也说明了这类游戏里，生存、资源、节奏控制往往比纯攻击更重要。

对当前项目来说，最值得借鉴的是：

- 用阶段表控制 AI 节奏，而不是让它每帧重新“思考整局”。
- 把炸弹/蓄力做成保命阈值，而不是纯进攻按钮。
- 允许少量随机性，但别破坏可解释的主行为。
