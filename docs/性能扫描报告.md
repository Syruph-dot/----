# 性能扫描报告

## 扫描范围
- 本次为静态代码审计，重点查看主循环、实体更新、碰撞路径、AI 决策、刷怪逻辑、渲染热路径，以及数组分配和 DOM / canvas 工作。
- 结论只基于代码结构与调用频率判断，未做运行时 profile。

## 结论摘要
- P0：AI 决策路径是最明显的 CPU 热点，尤其是困难模式。
- P0：碰撞、力场和帧末过滤存在明显的 O(B×E) 扫描与重复分配。
- P1：每帧渲染路径有较重的 canvas 开销，且重复扫描整批实体两次。
- P1：getEnemies() / getBullets() 返回新数组，带来额外 GC 压力。
- P2：刷怪、简单状态系统和初始化 DOM 不是当前主瓶颈。

## 确认热点

### 1. AI 决策搜索过重
- 位置： [src/systems/AIController.ts](../src/systems/AIController.ts#L123), [src/systems/AIController.ts](../src/systems/AIController.ts#L127), [src/systems/AIController.ts](../src/systems/AIController.ts#L287), [src/systems/AIController.ts](../src/systems/AIController.ts#L452), [src/systems/AIController.ts](../src/systems/AIController.ts#L510), [src/systems/AIController.ts](../src/systems/AIController.ts#L549), [src/systems/AIController.ts](../src/systems/AIController.ts#L608)
- 证据：update 以 decisionInterval 驱动决策；困难模式的间隔是 48ms。chooseMovementPlan() 会枚举多个候选 X / Y 位置，再对每个候选重复计算威胁与机会分数。computeThreatScore() 会再次遍历子弹和敌机；computeBulletThreat() 在困难模式下最多前瞻 46 帧，computeEnemyBodyThreat() 也会做多帧预测。
- 影响：这是明确的 CPU 放大器。敌弹越多、敌机越多、难度越高，决策开销越容易抢占帧预算。
- 风险：低到中。优化必须保持决策输出语义不变，尤其是技能请求顺序和移动策略。
- 说明：这是结构上就能确认的热点，不需要运行时数据也能判断其扩展性较差。

### 2. 碰撞、力场和帧末清理存在多层遍历
- 位置： [src/core/Game.ts](../src/core/Game.ts#L567), [src/core/Game.ts](../src/core/Game.ts#L591), [src/core/Game.ts](../src/core/Game.ts#L642), [src/core/Game.ts](../src/core/Game.ts#L950), [src/core/Game.ts](../src/core/Game.ts#L1061)
- 证据：update 先更新敌机和子弹，再进入 checkCollisions()；随后又执行子弹和敌机的 filter 清理。updateExpandingFields() 对每个力场再次扫描所有子弹和敌机，并在结束时对力场数组做两次 filter。checkCollisions() 本身又是子弹对敌机、子弹对 Boss、敌机对玩家的多层遍历。
- 影响：当子弹密度和技能效果同时上升时，这条路径会非常容易出现帧时间尖峰，并且每帧都在制造新数组。
- 风险：中。这里任何改动都不能改变命中顺序、击杀判定、力场消弹、传送子弹和连击结算。
- 说明：这是确认热点，且属于战斗核心语义区，优化时要优先保守。

### 3. 每帧渲染路径有明显 canvas 成本
- 位置： [src/core/Game.ts](../src/core/Game.ts#L647), [src/core/Game.ts](../src/core/Game.ts#L666), [src/core/Game.ts](../src/core/Game.ts#L700), [src/core/Game.ts](../src/core/Game.ts#L892)
- 证据：render 每帧都会画左右两个战区、分隔带和 UI。renderSideWorld() 每次都遍历整批敌机和子弹，再按 side 过滤；等价于每帧把同一批实体扫两遍。renderArenaBackdrop() 每帧重画网格、扫描线和边框，并且包含渐变、阴影和多次 stroke。renderScreenDivider() 也会每帧创建渐变并画刻度线。
- 影响：这是主要的 GPU / canvas 压力来源之一，尤其在实体数量中等以上时会放大。
- 风险：低到中。属于视觉层优化，通常不应改变游戏语义，但要注意别引入闪烁或层级错误。
- 说明：与 AI / 碰撞相比，这条路径更偏“固定帧开销”，在低实体数时也会持续消耗。

### 4. 过滤式访问造成额外分配
- 位置： [src/core/Game.ts](../src/core/Game.ts#L1379), [src/core/Game.ts](../src/core/Game.ts#L1383), [src/core/Player.ts](../src/core/Player.ts#L516), [src/core/Player.ts](../src/core/Player.ts#L519), [src/systems/AIController.ts](../src/systems/AIController.ts#L261), [src/systems/AIController.ts](../src/systems/AIController.ts#L551), [src/systems/WaveSystem.ts](../src/systems/WaveSystem.ts#L94)
- 证据：getEnemies() / getBullets() 每次都用 filter 返回新数组。AI、玩家炸弹、刷怪限制判断都会调用这些接口。update 末尾也会对活跃实体做过滤清理。
- 影响：单次开销不一定最大，但它会在多个高频路径上叠加，形成可见的 GC 压力。
- 风险：低。只要保持返回内容和调用顺序不变，就可以用缓存视图或原地压缩替换。
- 说明：这是比较典型的“重复扫描 + 重复分配”问题，属于值得优化的结构性热点。

## 可能热点

### 1. 子弹渲染与转移状态在大弹幕下会放大成本
- 位置： [src/core/Bullet.ts](../src/core/Bullet.ts#L93), [src/core/Bullet.ts](../src/core/Bullet.ts#L285), [src/core/Bullet.ts](../src/core/Bullet.ts#L358), [src/core/Bullet.ts](../src/core/Bullet.ts#L455)
- 判断：Bullet.render() 在不同状态下有多条分支，并且会使用 save() / restore()、渐变、阴影和路径绘制。update 还处理激光、线段激光、转移动画和命中缓存。
- 结论：在弹幕很多时它会变成明显负担，但是否超越 AI / 碰撞仍要看实际场景分布。

### 2. 波次生成的瞬时脉冲开销
- 位置： [src/systems/WaveSystem.ts](../src/systems/WaveSystem.ts#L29), [src/systems/WaveSystem.ts](../src/systems/WaveSystem.ts#L69), [src/systems/WaveSystem.ts](../src/systems/WaveSystem.ts#L94), [src/systems/WaveSystem.ts](../src/systems/WaveSystem.ts#L160)
- 判断：刷怪不是常驻热路径，但生成波次时会计算当前活跃敌机数量并创建多只敌机。这个成本更像短脉冲，不像持续性瓶颈。
- 结论：优先级低于主循环、AI 和渲染，但在极端刷怪节奏下仍值得留意。

## 非问题
- ComboSystem、ChargeSystem 和 ScoreSystem 都是简单的数值状态更新，不是主瓶颈。
- Enemy.update()、Boss.update() 的主体是线性移动和低频射击，不具备明显的算法放大效应。
- 初始化阶段的 DOM 构建属于一次性成本，不在帧热点之内。
